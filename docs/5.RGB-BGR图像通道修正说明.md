# RGB/BGR 图像通道修正说明

## 背景

本仓库的环境观测约定是：从 ROS2 相机话题得到的彩色图像在 `ARXEnv` 及其下游数据流中都保持为 **RGB `uint8`**。

OpenCV 的显示、图片编码、视频写入接口通常按 **BGR** 解释三通道彩色图像。因此本仓库现在采用统一规则：

- 环境观测、数据采集、LeRobot/PIL 输出：使用 RGB。
- `cv2.imwrite`、`cv2.imencode`、`cv2.VideoWriter.write`、`cv2.imshow` 等 OpenCV 可视化/编码入口：调用前把 RGB 转成 BGR。
- 深度图不做 RGB/BGR 通道转换，只在需要写视频预览时转成灰度 BGR 可视化帧。

## 原问题

问题出现在 ROS2 图像解码入口：

- `arx_toolkit/env/_ros2_io.py`
- `RobotIO._decode()`

原代码使用：

```python
img = self.bridge.imgmsg_to_cv2(msg, desired_encoding="passthrough")
if "depth" not in key:
    img = img[:, :, ::-1]  # BGR -> RGB
```

但实际通过该 ROS2 相机链路拿到的彩色图像已经是 RGB。这里再次执行 `[:, :, ::-1]` 会把 RGB 错误翻成 BGR，导致 `ARXEnv` 输出的 `{cam}_color` 与文档中标注的 RGB 语义不一致。

这个错误会继续影响：

- 机器人环境观测 `ARXEnv.get_observation()` 返回的相机图像。
- 数据采集写入 Zarr 的 `rgb_camera_*` 数据。
- LeRobot bridge 发送给策略的图像。
- Web 控制页面的实时 JPEG 预览。
- `_ros2_io` 中连续保存图片/视频时对 OpenCV 输入格式的判断。

## 修正内容

### 1. ROS2 解码保持 RGB

文件：`arx_toolkit/env/_ros2_io.py`

修改 `RobotIO._decode()`，删除非深度图的通道反转：

```python
img = self.bridge.imgmsg_to_cv2(msg, desired_encoding="passthrough")
```

现在 `_decode()` 对彩色图像的职责只是把 ROS 图像消息转成 numpy，并按需 resize；不会再假设输入是 BGR。

### 2. OpenCV 保存 PNG 前转 BGR

文件：`arx_toolkit/env/_ros2_io.py`

`cv2.imwrite()` 会按 BGR 解释输入，因此彩色 PNG 保存前新增：

```python
frame_bgr = cv2.cvtColor(img, cv2.COLOR_RGB2BGR)
cv2.imwrite(base_path + ".png", frame_bgr, ...)
```

深度图仍然保存为 `.npy`，不做通道转换。

### 3. 连续保存 MP4 前转 BGR

文件：`arx_toolkit/env/_ros2_io.py`

`RobotIO._save_video_frame()` 现在明确执行：

```python
elif "depth" not in key and frame.ndim == 3 and frame.shape[2] == 3:
    frame = cv2.cvtColor(frame, cv2.COLOR_RGB2BGR)
```

这样 `VideoWriter.write()` 收到的是 BGR，而环境和保存队列中的彩色图仍保持 RGB。

### 4. Web MJPEG 预览编码前转 BGR

文件：`scripts/web_control_demo.py`

Web 控制 demo 从 `ARXEnv` 读取的 `camera_*_color` 现在视为 RGB，并在 `cv2.imencode(".jpg", ...)` 前转换：

```python
rgb = np.asarray(cam_obs[cam_key])
frame_bgr = cv2.cvtColor(rgb, cv2.COLOR_RGB2BGR)
ok, jpeg = cv2.imencode(".jpg", frame_bgr, ...)
```

这修正了旧注释里“通道已反，所以不再转换”的临时绕法。

## 已核对但无需修改的位置

### 数据采集器

文件：`arx_toolkit/collect/collector.py`

采集器把 `obs["{cam}_color"]` 作为 RGB 写入 Zarr：

```python
buffer[f"rgb_{cam}"].append(rgb.transpose(2, 0, 1)[None])
```

保存回放 MP4 时已经正确执行：

```python
vw.write(cv2.cvtColor(frame_rgb, cv2.COLOR_RGB2BGR))
```

因此这里符合新约定，无需修改。

### LeRobot 转换脚本

文件：`scripts/convert_to_lerobot_v3.py`

该脚本从 Zarr 读取 `rgb_*`，转回 `(H, W, C)` 后交给 `PIL.Image.fromarray()` 或 LeRobot。PIL/LeRobot 图像语义为 RGB，因此无需 BGR 转换。

### LeRobot bridge

文件：`arx_toolkit/lerobot_bridge/arx_robot.py`

bridge 直接把 `ARXEnv` 的 `{camera}_color` 输出给 LeRobot observation。由于环境输出现在恢复为 RGB，这里无需额外转换。

## 后续维护约定

新增图像路径时请按下面规则判断：

- 变量名为 `rgb`、Zarr key 为 `rgb_*`、环境 observation 为 `*_color` 时，默认内容应为 RGB。
- 只有调用 OpenCV 的显示/图片编码/视频编码入口前，才做 `cv2.cvtColor(rgb, cv2.COLOR_RGB2BGR)`。
- 不要在 ROS2 解码入口做无条件 RGB/BGR 翻转；如果未来相机驱动改变 encoding，应优先根据 ROS `msg.encoding` 做显式判断，而不是依赖注释或经验猜测。
