## 三、关键组件详细分析

### 3.1 RobotIO 节点（底层通信）

#### ROS2 Topics

| 方向 | Topic | 消息类型 | 用途 |
|------|-------|---------|------|
| Pub | `arm_cmd_l` | `RobotCmd` | 左臂命令 |
| Pub | `arm_cmd_r` | `RobotCmd` | 右臂命令 |
| Pub | `/ARX_VR_L` | `PosCmd` | 底盘+升降命令 |
| Sub | `arm_status_l` | `RobotStatus` | 左臂状态 |
| Sub | `arm_status_r` | `RobotStatus` | 右臂状态 |
| Sub | `body_information` | `PosCmd` | 底盘状态 |
| Sub | `/{cam}_namespace/{cam}/...` | `Image` | 相机图像 |

#### ROS2 消息定义

**RobotCmd（控制命令）：**
```
std_msgs/Header header
float64[6] end_pos      # x y z roll pitch yaw
float64[6] joint_pos    # 6个关节角
float64    gripper      # 夹爪开合
int64      mode         # 4=末端控制, 5=关节控制
```

**RobotStatus（状态反馈）：**
```
std_msgs/Header header
float64[6]  end_pos     # 末端位姿
float64[]   joint_pos   # 关节位置(7维, 含gripper)
float64[]   joint_vel   # 关节速度(7维)
float64[]   joint_cur   # 关节电流(7维)
```

**PosCmd（底盘/升降）：**
```
float64 height          # 升降高度
float64 chx/chy/chz     # 底盘速度(前后/左右/旋转)
float64 head_pit/yaw    # 头部俯仰/偏航
float64[6] tempFloatData  # wheel1=idx1, wheel2=idx2, wheel3=idx3
int32 mode1/mode2       # 控制模式
```

#### 相机系统关键设计

- 使用 `ApproximateTimeSynchronizer`，**slop=20ms**，多相机近似同步
- 后台 `_save_worker` 线程**异步写图片/视频**，不阻塞主循环
- 支持 `color` / `aligned_depth_to_color` 两种图像类型
- 默认3个视角：`camera_l`, `camera_h`, `camera_r`
- Topic 命名规则：`/{cam}_namespace/{cam}/color/image_rect_raw` 和 `/{cam}_namespace/{cam}/aligned_depth_to_color/image_raw`

### 3.2 ARXRobotEnv（环境主类）

#### 构造函数参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `duration_per_step` | 1/20 | 平滑插值单步时长（20Hz） |
| `min_steps` | 20 | 平滑轨迹最少插值步数 |
| `max_v_xyz` | 0.25 | 平移最大速度 |
| `max_v_rpy` | 0.3 | 姿态最大速度 |
| `max_a_xyz` | 0.3 | 平移最大加速度 |
| `max_a_rpy` | 0.8 | 姿态最大加速度 |
| `camera_type` | "all" | 图像类型: "color"/"depth"/"all" |
| `camera_view` | ("camera_l","camera_h","camera_r") | 订阅的相机列表 |
| `dir` | None | 默认保存目录 |
| `video` | False | 是否保存视频 |
| `video_fps` | 20.0 | 视频帧率 |
| `img_size` | (224, 224) | 图像缩放尺寸 |

#### 11个公开控制方法

| 方法 | 控制类型 | 说明 |
|------|---------|------|
| `step_smooth_eef(action)` | 末端+插值 | **最常用**，梯形速度曲线 + SLERP ｜不要这个了
| `step_raw_eef(action)` | 末端直发 | 无规划，直接发 mode=4 |这个是absolute eef控制模式
| `step_delta_eef(action)` | 末端增量 | 四元数旋转叠加，适合视觉伺服 |delta eef控制模式
| `step_smooth_joint(action)` | 关节+插值 | 线性插值逐步发 |不要这个
| `step_raw_joint(action)` | 关节直发 | 直接发 mode=5 |joint控制模式
| `step_delta_joint(action)` | 关节增量 | 基于当前位置叠加 |不要这个
| `step_lift(height)` | 升降 | 步进0.03逐步到位 |0-20（这个范围是0-20，控制升降台的高度，0是最低 20是最高）
| `step_base(vx,vy,vz)` | 底盘速度 | 单次发送，需手动停止 |-1.5-1.5 vx vy是两个方向想达到的速度 vz是旋转用-2 - 2
| `step_base_lift(vx,vy,vz,h)` | 底盘+升降 | 联合控制 |这个保留，是上两者的联合
| `set_special_mode(mode,side)` | 模式切换 | 0=soft, 1=home, 2=protect, 3=gravity |
| `reset()` / `close()` | 生命周期 | 初始化复位 / 安全退出 |

#### action 格式统一为字典

```python
action = {
    "left":  np.array([x,y,z,r,p,y,gripper], dtype=np.float32),  # 7维
    "right": np.array([x,y,z,r,p,y,gripper], dtype=np.float32),  # 7维
}
# 可以只传单臂
```

- 末端类接口：前6维 = `[x, y, z, roll, pitch, yaw]`，第7维 = `gripper`
- 关节类接口：前6维 = 6个关节角，第7维 = `gripper`

### 3.3 轨迹规划算法（核心数学）

#### 整体流程

```
输入: 当前观测 curr_obs + 目标 action
  ↓
1. 计算 xyz 位移距离 d_xyz（取各轴最大绝对值）
2. 计算 rpy 角距离 d_rpy（四元数空间 arccos）
  ↓
3. 分别做梯形速度规划
   xyz: _trapezoid_params(d_xyz, max_v_xyz, max_a_xyz) → 总时间 t_xyz
   rpy: _trapezoid_params(d_rpy, max_v_rpy, max_a_rpy) → 总时间 t_rpy
  ↓
4. 取较慢的那个决定总步数: max(steps_xyz, steps_rpy)
   且不低于 min_steps
  ↓
5. 逐步生成插值序列:
   - xyz: 梯形曲线映射 → 线性插值
   - rpy: 梯形曲线映射 → SLERP 四元数插值
   - gripper: 线性插值，在 50% 时间点完成
  ↓
输出: Dict[side, List[np.ndarray(7,)]]  逐步发送序列
```

#### 梯形速度曲线

```
速度 ^
     |    v_peak
     |   /------\
     |  /        \
     | /          \
     +---+--------+---→ 时间
       t_accel  t_flat  t_accel

当位移太短时退化为三角曲线（达不到 v_max）
```

#### SLERP 四元数插值

- 避免直接线性插值 RPY 带来的万向锁和姿态跳变
- `_quat_from_rpy` → `_quat_slerp` → `_rpy_from_quat`
- 当 dot > 0.9995 时退化为线性插值（近似共线）

#### Delta 控制的旋转叠加

```python
# delta_eef 中旋转增量用四元数乘法叠加
q_curr   = quat_from_rpy(curr_rpy)
q_delta  = quat_from_rpy(delta_rpy)
q_target = quat_multiply(q_delta, q_curr)  # 左乘 = 基坐标系下的增量
target_rpy = rpy_from_quat(q_target)
```

## 四、对构建模仿学习环境的评估

### 优势（可直接复用）

1. **双臂支持** — 天然的 leader-follower 遥操作基础，action 字典 left/right 设计良好
2. **观测空间完整** — joint_pos/vel/cur + 多相机 + 深度图，Pi0.5 需要的都有
3. **轨迹插值成熟** — 梯形速度曲线 + SLERP，动作平滑且有加速度约束
4. **相机同步** — ApproximateTimeSynchronizer 确保多视角时间对齐（±20ms）
5. **后台异步存储** — save_worker 线程不阻塞控制循环
6. **模式切换完善** — soft/home/protect/gravity 四种安全模式

### 需要改进/新增的部分

| 问题 | 现状 | 改进方案 |
|------|------|---------|
| **无 Gym 接口** | 不是标准 `gymnasium.Env`，无 `action_space`/`observation_space` 定义 | 包一层 Gym wrapper，定义 Box space |
| **无数据采集管线** | 只有单帧 PNG / 视频 MP4 保存，无结构化数据存储 | 需加 `DataRecorder` → Zarr → LeRobot v3 |
| **控制频率不固定** | `step_smooth_eef` 内部阻塞执行完所有插值步，外部循环无固定频率保证 | 需加固定频率控制循环（Timer-based） |
| **无遥操作模块** | 没有 leader-follower 映射 或 VR 输入代码 | 需单独实现 Teleop 模块 |
| **深耦合 ROS2** | 直接依赖 `rclpy`、`cv_bridge`、ROS msg，机器A无法运行 | 需抽象硬件层，支持 Mock 后端 |
| **无异步 action** | `step_smooth_eef` 是同步阻塞的，执行完才返回 | 对高频 VLA 推理不友好，需非阻塞版本 |
| **action_space 未定义** | `_setup_space()` 是 TODO | 需要正式定义 action/obs space |
| **无评测框架** | 没有成功率统计、任务计时等 | Deploy 阶段需要 eval runner |

### 关键设计决策

1. **hardware 抽象层** — 真机 vs Mock 通过同一接口切换，机器A可以跑完整流程
2. **gymnasium.Env 标准接口** — `obs, reward, terminated, truncated, info = env.step(action)`，兼容所有 VLA 框架
3. **固定频率控制循环** — `PolicyRunner` 以固定 Hz（如 20Hz）推理+发送，不受轨迹插值阻塞
4. **数据格式对齐 LeRobot v3** — 采集直接存储兼容格式，减少转换开销
5. **配置驱动** — YAML 配置文件管理所有硬件参数，方便切换不同机器人/相机

### 建议开发优先级

| 优先级 | 模块 | 理由 |
|--------|------|------|
| P0 | Env + Mock | 机器A能跑通基本流程，验证接口设计 |

| P2 | Collect | 数据落盘 + LeRobot v3 格式 |
